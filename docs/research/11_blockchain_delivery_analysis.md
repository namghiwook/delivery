# 블록체인/Web3 기반 배달 서비스 분석

## 1. 블록체인 배달 서비스 시도 사례

### 1.1 전 세계 시도 사례 총정리

블록체인을 배달 서비스에 직접 적용하려는 시도는 전 세계적으로 극히 드물며, 대부분 초기 단계에서 실패하거나 사실상 사라졌다.

#### (1) FoodCoin Ecosystem (FOOD)
- **시기**: 2017년 12월 ICO (12/12~12/26)
- **개요**: 이더리움 기반 블록체인 생태계. 1000EcoFarms 플랫폼을 기반으로 식품·농산물의 글로벌 마켓플레이스를 구축하겠다는 비전
- **토큰**: FOOD 토큰 10억 개 발행, 이더리움 ERC-20
- **현재 상태**: 사실상 사망(dead project). 작동하는 제품 없음. 팀/자문단 토큰에 대한 베스팅(vesting) 스케줄 없이 발행되어 레드플래그로 지적됨
- **실패 원인**: 실질적 서비스 없이 토큰만 발행. ICO 버블기(2017~2018)의 전형적 "백서만 있는 프로젝트"
- **출처**: [CryptoRank FoodCoin ICO 분석](https://cryptorank.io/ico/food), [TokenMarket - ICO Over](https://tokenmarket.net/blockchain/ethereum/assets/foodcoin/)

#### (2) Munchee (MUN) - SEC에 의해 중단
- **시기**: 2017년 11월 ICO 시작, 2017년 12월 SEC에 의해 중단
- **개요**: 캘리포니아 소재. 기존 음식 리뷰 아이폰 앱을 블록체인 기반 생태계로 확장하려 함. $15M 자금 조달 목표
- **토큰**: MUN 토큰
- **중단 경위**: SEC가 Howey 테스트를 적용, MUN 토큰이 증권(securities)에 해당한다고 판단. 미등록 증권 판매로 규정. Munchee는 토큰 배포 전 ICO를 중단하고 투자자에게 전액 환불
- **역사적 의의**: 직접적 사기 혐의 없이 SEC가 ICO를 중단시킨 최초 사례. 이후 모든 음식/배달 관련 토큰 프로젝트에 대한 규제 경고로 작용
- **출처**: [SEC 공식 보도자료](https://www.sec.gov/newsroom/press-releases/2017-227), [CNN Money](https://money.cnn.com/2017/12/11/news/munchee-initial-coin-offering-sec/index.html)

#### (3) 한국: 코인플러그 "머글시간" + 마이키핀(MYKEEPiN)
- **시기**: 2021년 한국인터넷진흥원(KISA) 블록체인 시범사업으로 선정
- **개요**: 코인플러그와 딜리온그룹이 공동 개발. 블록체인 DID(분산ID) 기반 배달앱 "머글시간"과 DID 인증앱 "마이키핀"을 연동. 경남 김해시에서 시범 운영
- **적용 범위**: 블록체인을 배달 매칭이나 결제 자체에 적용한 것이 아니라, 본인인증(DID)에 블록체인을 활용한 것. W3C DID 표준 준수
- **현재 상태**: 시범사업 이후 상용 확산에 대한 후속 정보를 찾을 수 없음. 사실상 시범사업 단계에서 중단된 것으로 추정
- **출처**: [CPLABS Medium - O2O 블록체인 적용사례](https://medium.com/cplabs-official/o2o-%EC%83%9D%ED%83%9C%EA%B3%84-%EB%B0%B0%EB%8B%AC-%ED%94%8C%EB%9E%AB%ED%8F%BC%EC%97%90%EC%84%9C%EC%9D%98-%EB%B8%94%EB%A1%9D%EC%B2%B4%EC%9D%B8-%EA%B8%B0%EC%88%A0-%EC%B2%AB-%EC%A0%81%EC%9A%A9%EC%82%AC%EB%A1%80-%EC%86%8C%EA%B0%9C-25dc3f8f6997)

#### (4) 한국: AoneA 코인 + 쏜다 배달대행
- **시기**: 정확한 시작일 불명, 2020년대 초반 추정
- **개요**: (주)엔베스트가 운영. Klaytn(클레이튼) 블록체인 기반 KIP-7 토큰 A1A 발행. 배달 운송의 효율적 비용 산정과 AI 경로 최적화를 블록체인 빅데이터와 연계하겠다는 비전
- **현재 상태**: A1A 토큰이 해외 거래소에 상장 시도 중이나 거래량 극히 미미. 실질적 사용자 기반에 대한 정보 부족. 사실상 토큰 프로젝트로서의 성격이 강함
- **출처**: [BlockchainUS 보도](https://www.blockchainus.co.kr/news/articleView.html?idxno=1560), [CoinLore A1A](https://www.coinlore.com/coin/aonea-coin)

#### (5) 학술적 제안/프로토타입 (실제 서비스 아님)
아래는 논문 수준에서 제안된 블록체인 배달 시스템으로, 실제 상용 서비스로 출시된 적 없음:
- **FoodBlock**: 이더리움 기반 온라인 식품 주문 프레임워크 (2025년 학술 논문)
- **P2P Smart Food Delivery Platform**: 스마트 컨트랙트 기반 P2P 배달 플랫폼 (2022년 MDPI 논문)
- **Decentralized Food Delivery for Smart Cities**: 블록체인 + IoT 모델 (2023년 Springer 논문)
- **출처**: [Wiley - FoodBlock](https://onlinelibrary.wiley.com/doi/10.1111/exsy.13601), [MDPI - P2P Smart Food Delivery](https://www.mdpi.com/2079-9292/11/12/1806)

### 1.2 관련 대형 실패 사례: TradeLens (물류 블록체인)

배달은 아니지만, 물류 영역에서 가장 큰 블록체인 실패 사례로 교훈이 있다.

- **주체**: Maersk(세계 최대 해운사) + IBM
- **시기**: 2018년 출범 → 2022년 폐쇄
- **규모**: 수억 달러 투자 추정
- **실패 원인**:
  - 경쟁사들이 Maersk가 주도하는 플랫폼에 데이터를 공유하길 거부
  - 아시아/중국 해운사 어느 곳도 참여하지 않음
  - 네트워크 효과를 달성하지 못함 ("모든 참여자가 win-win 상태에 있고, 명확한 ROI가 있을 때만 블록체인 프로젝트가 성공한다" - Gartner)
  - "제품을 먼저 만들고 고객이 오길 기대하지 마라. 고객이 준비된 제품을 만들어라"
- **핵심 교훈**: 기술이 아무리 좋아도, 네트워크 참여자 확보 없이는 블록체인 플랫폼은 의미 없음
- **출처**: [Supply Chain Dive](https://www.supplychaindive.com/news/Maersk-IBM-shut-down-TradeLens/637580/), [CoinTelegraph](https://cointelegraph.com/news/the-ibm-maersk-blockchain-effort-was-doomed-to-fail-from-the-start)

## 2. 왜 블록체인 배달 서비스가 존재하지 않는가?

### 2.1 구조적 부적합성

#### (1) 실시간 매칭/배차와 블록체인의 근본적 충돌

배달 서비스의 핵심은 **초 단위 실시간 의사결정**이다:
- 주문 접수 → 가맹점 전달 → 라이더 배차 → 경로 최적화가 수 초 내에 이루어져야 함
- 배민, 쿠팡이츠는 AI 기반 실시간 배차 알고리즘으로 밀리초 단위 의사결정 수행

블록체인의 본질적 특성과의 충돌:
| 항목 | 배달 서비스 요구사항 | 블록체인 특성 |
|------|---------------------|---------------|
| 속도 | 밀리초~초 단위 응답 | 블록 생성 시간 필요 (이더리움 ~12초, L2도 수초) |
| 비용 | 주문 1건당 추가비용 최소화 | 가스비 발생 (L2 기준 $0.01~$0.20) |
| 데이터 수정 | 주문 변경, 취소 빈번 | 불변성(immutability)이 핵심 가치 |
| 프라이버시 | 개인정보 보호 필수 | 공개 원장이 기본 |
| 처리량 | 피크 시간 대량 주문 동시 처리 | TPS 제한 (L2 최대 1,000~4,000 TPS) |

**결론**: 블록체인은 "모든 참여자가 검증하는 합의"가 필요한 구조이므로, 중앙 서버보다 본질적으로 느리고 비싸다. 실시간 배차에 블록체인을 쓰는 것은 "못을 박는데 현미경을 쓰는 것"과 같다.

#### (2) 네트워크 효과의 역설

블록체인의 가치는 탈중앙화에서 나오지만, 배달 서비스의 가치는 **네트워크 밀도**에서 나온다:
- 한 동네에 가맹점 100개, 라이더 50명이 있어야 서비스가 작동
- 이러한 밀도를 탈중앙 방식으로 달성하는 것은 치킨-앤-에그 문제의 극대화
- TradeLens 실패가 보여주듯, 참여자 확보 없는 블록체인 플랫폼은 무의미

#### (3) 소비자 경험과의 괴리

- 일반 소비자는 "배달 앱을 켜고 → 음식을 고르고 → 결제하고 → 기다린다"
- 이 과정에서 "블록이 생성되고 있습니다, 잠시 기다려주세요"라는 경험은 치명적
- 지갑 설정, 토큰 구매, 가스비 이해 등은 대중 서비스에 부적합
- 배달의민족 사용자 수천만 명 중 암호화폐 지갑을 가진 사용자 비율은 극소수

### 2.2 스마트 컨트랙트 정산 자동화의 가능성과 한계

#### 가능한 이점
- **즉시 정산**: 현재 배민 정산 주기(D+2~D+7)를 실시간으로 전환 가능
- **투명한 수수료**: 스마트 컨트랙트에 수수료율이 코드로 공개되어 점주가 검증 가능
- **중간 정산 사업자 제거**: 플랫폼 → PG → 은행 → 점주로 이어지는 정산 체인 단순화

#### 현실적 한계
- 즉시 정산은 블록체인 없이도 가능 (핀테크 선정산 서비스 이미 존재)
- 원화 기반 결제를 토큰으로 전환하는 순간 "가상자산사업자" 규제 적용
- 2025년 배민 수수료 논란(점주 최종 정산 55%까지 차감)은 기술이 아닌 시장 지배력의 문제

### 2.3 토큰 인센티브의 이론과 현실

#### 이론적 가능성
- 점주: 토큰 보유 시 수수료 할인, 거버넌스 참여
- 소비자: 주문 시 토큰 리워드, 리뷰 작성 보상
- 라이더: 배달 완료 시 즉시 토큰 정산, 우수 서비스 보너스

#### 현실적 문제
- 토큰 가치 변동성 → 점주 매출이 토큰 가격에 연동되는 리스크
- "포인트로도 할 수 있는 것"을 왜 토큰으로 해야 하는가?
- 토큰 설계 실패 시 투기 수단으로 전락 → 서비스 본질 훼손

## 3. 만약 블록체인 배달 플랫폼을 만든다면?

### 3.1 토큰 이코노미 설계

#### 이중 토큰 구조 제안

**a) 유틸리티 토큰 (예: DLVR)**
- **용도**: 배달비 결제, 수수료 할인, 리워드
- **특징**: 스테이블코인에 페깅(pegging)하거나, 원화 연동 가치 안정 메커니즘 필수
- **발행량**: 고정 공급 또는 소각(burn) 메커니즘으로 인플레이션 통제
- **수수료 할인**: DLVR로 결제 시 수수료 2~3% 할인 (플랫폼 사용 인센티브)

**b) 거버넌스 토큰 (예: gDLVR)**
- **용도**: DAO 투표권, 정책 결정 참여
- **획득 방법**: 플랫폼 활동(주문, 배달, 리뷰) 기반 에어드롭. 구매 불가(투기 방지)
- **투표 범위**: 수수료율 변경, 신규 기능 도입, 분쟁 해결 정책

#### 토큰 가치 유지 메커니즘
1. **수수료 소각**: 플랫폼 수수료의 일부를 DLVR 매입 후 소각
2. **스테이킹 보상**: DLVR 스테이킹 시 거버넌스 토큰 보상
3. **실 사용 연동**: 토큰의 가치를 투기가 아닌 플랫폼 내 실사용에 연동
4. **가격 안정 풀**: 알고리즘 마켓메이커(AMM) 풀로 급격한 가격 변동 완화

#### 참고 사례: Visa의 스테이블코인 파일럿
- 2025년 11월, Visa가 USDC 스테이블코인으로 긱 워커(gig workers)에게 직접 정산하는 파일럿 시작
- 크로스보더 결제에서 며칠 걸리던 정산을 분 단위로 단축
- **시사점**: 배달 라이더 정산에 스테이블코인을 적용하면 D+7을 즉시 정산으로 전환 가능
- **출처**: [Visa 공식 발표](https://investor.visa.com/news/news-details/2025/Visa-Direct-Stablecoin-Payouts-Pilot-Speeds-Up-Access-to-Funds-for-Creators--Gig-Workers/default.aspx), [Bloomberg](https://www.bloomberg.com/news/articles/2025-11-12/visa-tests-usdc-stablecoin-payouts-targeting-global-gig-workers-influencers)

### 3.2 DAO(탈중앙 자율 조직) 거버넌스

#### 구조 설계

```
┌─────────────────────────────────────────────────┐
│                  배달 DAO                         │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │ 점주 위원회 │  │ 소비자 위원회│  │ 라이더 위원회│       │
│  │ (33% 투표권)│  │ (33% 투표권)│  │ (34% 투표권)│       │
│  └──────────┘  └──────────┘  └──────────┘       │
│                                                   │
│  투표 대상:                                        │
│  - 중개 수수료율 (현재 2.0~7.8% → DAO 결정)        │
│  - 배달비 구조 (점주 부담 vs 소비자 부담 비율)       │
│  - 프로모션/할인 정책                               │
│  - 라이더 최저 배달료                               │
│  - 분쟁 해결 정책                                   │
│                                                   │
│  실행: 스마트 컨트랙트로 자동 반영                    │
└─────────────────────────────────────────────────┘
```

#### 현실적 가능 여부 분석

**긍정적 측면:**
- The Drivers Cooperative (뉴욕, 2021년 출범) 사례: 라이더 소유 협동조합 방식으로 Uber/Lyft 대비 8~10% 높은 라이더 수익, 15% 수수료(Uber 25~40% 대비). 2,500명 이상 참여. 1인 1표 민주적 의사결정
- 이미 "협동조합형 플랫폼"이 작동함을 증명. DAO는 이를 기술적으로 확장하는 도구
- **출처**: [The Drivers Cooperative](https://drivers.coop/), [Wikipedia](https://en.wikipedia.org/wiki/The_Drivers_Cooperative)

**부정적 측면:**
- 투표 참여율 문제: 대부분의 DAO에서 투표 참여율은 5~15%에 불과
- 의사결정 속도: 배달 시장은 빠른 대응이 필요 (프로모션, 경쟁 대응 등), DAO 투표는 수일~수주 소요
- 전문성 부족: 수수료율 결정에는 재무/경영 전문성이 필요. 모든 참여자의 다수결이 최선인가?
- 규제 충돌: DAO 자체가 한국 법제에서 "법인격 없는 단체"로 인정될 수 있는가?

**현실적 제안: 하이브리드 모델**
- 일상 운영: 전문 경영팀이 중앙화 방식으로 운영
- 핵심 정책: 수수료율, 배달비 구조 등 핵심 정책만 DAO 투표로 결정
- 분기별 거버넌스: 매일 투표가 아닌, 분기별 주요 정책 투표

### 3.3 스마트 컨트랙트 활용

#### 주문~정산 프로세스 자동화

```
[주문] ──→ [결제 에스크로] ──→ [조리 확인] ──→ [라이더 픽업] ──→ [배달 완료] ──→ [자동 정산]
  │            │                  │               │                │              │
  │     소비자 결제금            점주 확인        GPS 확인       수령 확인      즉시 분배
  │     스마트 컨트랙트         오라클 연동     IoT/GPS 연동   소비자 확인     점주: 83%
  │     에스크로 잠금                                                          라이더: 12%
  │                                                                           플랫폼: 5%
```

#### 구체적 활용 시나리오

| 단계 | 현행 방식 | 스마트 컨트랙트 방식 | 개선점 |
|------|----------|---------------------|--------|
| 결제 | PG사 통해 카드 결제 | 스테이블코인 에스크로 | 정산 수수료 절감 |
| 조리 | 점주 앱에서 수동 확인 | IoT 센서/점주 확인 → 오라클 | 데이터 위변조 방지 |
| 배달 | 플랫폼 GPS 추적 | GPS 오라클 → 컨트랙트 검증 | 투명한 배달 기록 |
| 정산 | D+2~D+7 일괄 정산 | 배달 완료 즉시 자동 분배 | **즉시 정산** |
| 분쟁 | 고객센터 수동 처리 | 조건부 에스크로 + 투표 | 자동/반자동 분쟁 해결 |

#### 분쟁 해결 자동화

```
배달 완료 후 소비자 불만 접수
    │
    ├── 배달 지연 (GPS 데이터로 자동 판단)
    │   └── 30분 이상 지연 시 배달비 자동 환불
    │
    ├── 음식 누락/파손 (사진 증거 + AI 판독)
    │   └── 증거 확인 시 에스크로에서 자동 부분 환불
    │
    └── 분쟁 (당사자 간 이견)
        └── 중재 위원회(DAO 선출) 투표로 결정
            └── 중재 결과 스마트 컨트랙트로 자동 집행
```

### 3.4 현실적 한계

#### (1) 한국 법규 제약

**특정금융정보법(특금법)**:
- 가상자산사업자로 금융정보분석원(FIU)에 신고 필수
- ISMS 인증 + 실명 확인 입출금 계정 필수
- 미신고 영업 시 형사처벌 대상
- **의미**: 배달앱이 토큰을 발행·유통하려면 "가상자산사업자"로 별도 신고해야 하며, 이는 배달 사업과 전혀 다른 규제 영역

**가상자산 이용자 보호법 (2024년 7월 시행)**:
- 이용자 자산 보호 의무 강화
- 불공정거래 규제 본격 도입
- **의미**: 토큰 가격 조작, 미공개 정보 이용 등에 대한 엄격한 규제. 배달 플랫폼이 자체 토큰을 운영하면 이 법의 적용 대상

**전자금융거래법**:
- 토큰으로 결제를 처리하면 "전자금융업자" 등록 필요 가능성
- 원화↔토큰 환전 서비스는 추가 인허가 필요

**출처**: [김앤장 - 특금법 시행령 개정안](https://www.kimchang.com/ko/insights/detail.kc?sch_section=4&idx=22391), [법제처 - 가상자산이용자보호법](https://www.law.go.kr/LSW/lsInfoP.do?lsiSeq=261099)

#### (2) 소비자 진입장벽

- 암호화폐 지갑 설치 및 관리
- 토큰 구매 과정 (거래소 가입 → KYC → 원화 입금 → 토큰 교환)
- 가스비 개념 이해
- 프라이빗 키 관리 (분실 시 자산 영구 소실)

**완화 방안**: 계정 추상화(Account Abstraction, ERC-4337), 소셜 로그인 지갑 등으로 사용자 경험을 개선할 수 있으나, 아직 대중화 단계는 아님.

#### (3) 블록체인 트랜잭션 속도/비용

2025년 기준 Layer 2 성능:
| 네트워크 | TPS | 평균 수수료 |
|---------|-----|-----------|
| Base | 1,000+ | $0.01 |
| Polygon | 700 | $0.05 |
| Arbitrum | 400 | $0.20 |
| Optimism | 300 | $0.15 |

- **긍정적**: L2 기술 발전으로 비용/속도 문제는 상당히 해소됨
- **부정적**: 여전히 중앙 서버(수만 TPS, 수수료 $0)보다 느리고 비쌈
- **출처**: [PayRam - L2 비교](https://www.payram.com/blog/arbitrum-vs-optimism-vs-base)

#### (4) "블록체인이 아니어도 가능한 것" 문제

현재 배달 시장의 문제와 블록체인의 필요성을 정직하게 대조하면:

| 문제 | 블록체인 해결? | 블록체인 없이 해결 가능? |
|------|-------------|---------------------|
| 높은 수수료 | 탈중앙화로 중개자 제거 | 공공배달앱(배달특급), 협동조합 모델 |
| 늦은 정산 | 스마트 컨트랙트 즉시 정산 | 핀테크 선정산 서비스 (토스, 세틀뱅크 등) |
| 불투명한 수수료 | 온체인 투명성 | 정부 규제, 공시 의무 |
| 데이터 독점 | 탈중앙 데이터 소유 | 데이터 이동권 법제화 (마이데이터) |
| 리뷰 조작 | 온체인 불변 리뷰 | AI 기반 리뷰 검증 시스템 |
| 라이더 처우 | DAO 거버넌스 | 노동법 강화, 협동조합 |

**솔직한 평가**: 위 문제 중 블록체인이 "유일한 해법"인 것은 거의 없다.

## 4. 블록체인이 배달에서 진짜 가치를 만들 수 있는 지점

### 4.1 블록체인이 아니면 불가능한 것

엄밀히 말해, 배달 서비스에서 "블록체인이 아니면 절대 불가능한 것"은 거의 없다. 하지만 블록체인이 **가장 효율적인 도구**일 수 있는 영역은 존재한다:

#### (1) 크로스-플랫폼 라이더 평판/이력 시스템
- 라이더가 배민, 쿠팡이츠, 요기요 등 여러 플랫폼에서 쌓은 배달 이력과 평판을 하나의 온체인 프로필로 통합
- 플랫폼 전환 시에도 평판이 유지되는 "포터블 평판"
- 이는 중앙화 시스템으로는 플랫폼 간 협력이 필요하므로 구현 어려움

#### (2) 스테이블코인 기반 즉시 정산
- Visa의 USDC 파일럿이 보여주듯, 스테이블코인 즉시 정산은 현실화 단계
- 라이더에게 배달 완료 즉시 USDC/USDT로 정산 → 은행 계좌로 즉시 환전
- 기존 PG → 은행 정산 체인의 수수료와 지연을 완전히 제거
- **이것은 "블록체인 배달앱"이 아닌 "블록체인 결제 인프라"를 배달에 접목하는 것**

#### (3) 소상공인 간 탈중앙 주문 네트워크
- 점주들이 배민 없이 직접 주문을 받을 수 있는 오픈 프로토콜
- 누구나 프론트엔드(앱)를 만들 수 있고, 주문 데이터는 공유 블록체인에 기록
- "배달 프로토콜"로서의 블록체인 활용 (HTTP가 웹의 프로토콜인 것처럼)
- 이론적으로 가능하나, 네트워크 효과 확보가 극히 어려움

### 4.2 투명성 / 탈중앙화 / 토큰 인센티브 중 실질적 가치

#### 투명성: 가치 있지만 블록체인 필수는 아님
- 배민의 수수료 구조가 불투명하다는 점주 불만은 실재
- 그러나 정부 규제와 공시 의무로도 해결 가능 (2024년 상생협의체 사례)
- 블록체인의 온체인 투명성은 기술적으로 우월하나, 채택 비용 대비 효과 불확실

#### 탈중앙화: 가장 큰 가치, 그러나 가장 큰 도전
- 배민(점유율 59%), 쿠팡이츠(24%)의 독과점이 시장의 근본 문제
- 탈중앙 플랫폼은 이론적으로 "플랫폼 없는 플랫폼"을 가능하게 함
- **그러나**: 탈중앙 플랫폼이 중앙화 플랫폼의 UX, 마케팅, 네트워크 효과를 이기기 극히 어려움
- The Drivers Cooperative 같은 "인간적 탈중앙화(협동조합)"가 기술적 탈중앙화(블록체인)보다 현실적

#### 토큰 인센티브: 양날의 검
- 초기 사용자 확보에 효과적일 수 있음 (에어드롭, 토큰 리워드)
- 그러나 "토큰 가격이 올라가니까 쓴다" → "토큰 가격이 떨어지니 이탈" 패턴 반복
- 포인트 대비 토큰의 진짜 추가 가치는 "2차 시장에서의 거래 가능성" 뿐
- 이 거래 가능성이 투기를 불러 서비스 본질을 훼손할 위험

### 4.3 솔직한 결론

#### 블록체인 배달은 현실적으로 의미가 있는가?

**단기적(1~3년): 아니다.**
- 블록체인 자체를 배달 서비스의 핵심 인프라로 쓰는 것은 과잉 기술(over-engineering)
- "배달 문제"는 기술 문제가 아닌 시장 구조 문제(독과점, 수수료, 라이더 처우)
- 이 문제의 해결책은 규제, 협동조합, 공공 플랫폼이 블록체인보다 현실적

**중기적(3~5년): 부분적으로 의미 있다.**
- **정산 레이어**: 스테이블코인 기반 즉시 정산은 Visa 파일럿에서 보듯 현실화 중
- **신원/평판**: DID 기반 라이더 포터블 평판은 기술적으로 가능하며 가치 있음
- **데이터 주권**: 소비자/점주의 데이터를 플랫폼이 아닌 본인이 소유하는 모델

**장기적(5~10년): 불확실하지만 주목할 만하다.**
- Layer 2/3 기술의 발전으로 속도/비용 문제가 해소되면
- 계정 추상화 등으로 사용자 경험이 "보이지 않는 블록체인"이 되면
- "배달 프로토콜"로서의 탈중앙 인프라가 가능할 수 있음
- 그러나 이는 배달만의 문제가 아닌 웹3 전체 생태계의 성숙도에 의존

#### 가장 현실적인 접근: 하이브리드 모델

```
┌─────────────────────────────────────────────────────────┐
│              현실적 블록체인 배달 모델                      │
│                                                           │
│  [프론트엔드]     일반 앱 (배민과 동일한 UX)               │
│  [비즈니스로직]   중앙 서버 (실시간 매칭, AI 배차)         │
│  [정산 레이어]    ★ 스마트 컨트랙트 (스테이블코인 즉시 정산)│
│  [신원/평판]      ★ DID 기반 라이더 포터블 프로필          │
│  [거버넌스]       ★ 핵심 정책만 DAO 투표 (분기별)          │
│  [데이터]         ★ 주문 해시를 온체인에 기록 (증명용)      │
│                                                           │
│  ★ = 블록체인 적용 영역 (전체의 20~30%)                    │
│  나머지 70~80%는 기존 기술로 운영                          │
└─────────────────────────────────────────────────────────┘
```

**핵심 메시지**: "블록체인 배달앱"을 만드는 것이 아니라, 배달앱의 **가장 아픈 지점**(높은 수수료, 느린 정산, 플랫폼 독점)에 블록체인 기술을 **외과적으로 적용**하는 것이 유일하게 현실적인 접근이다.
